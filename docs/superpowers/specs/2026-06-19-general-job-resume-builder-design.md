# General Job Resume Builder 设计文档

## 1. 概述

当前项目已经完成校招简历编辑器 MVP，具备：

- 结构化填写
- 实时预览
- 模块拖拽排序
- 本地保存
- 打印导出 PDF
- 本地 JSON 备份与恢复

下一阶段不再把产品限定为“校招简历编辑器”，而是升级为：

`通用求职简历工具`

目标是覆盖学生、职场人和不确定自身分类的通用求职者，同时保持产品当前的低门槛、稳定导出和桌面端优先体验。

## 2. 目标与边界

### 2.1 产品目标

- 扩大适用人群，从校招扩展到更广的求职场景
- 继续保持“好上手、好预览、好导出”的核心体验
- 用统一编辑器内核承载不同人群的默认起点
- 通过官方模块库扩展常见求职信息结构

### 2.2 非目标

- 不拆成学生版、职场版、通用版三个独立产品
- 不实现任意自定义模块类型
- 不扩成个人主页、作品集站或多用途资料平台
- 不偏离正式求职简历 PDF 这一核心交付物
- 不在本阶段引入账号、云同步或后端存储

### 2.3 核心原则

- 一个编辑器内核
- 身份轻分流
- 官方模块库扩展
- 安全切换身份
- 所有模块都必须适合进入正式求职简历 PDF

## 3. 目标用户

第一阶段覆盖三类用户入口：

- `学生`
- `职场人`
- `通用求职者`

三者差异主要体现在：

- 默认模块配置
- 默认排序
- 填写引导文案

而不是底层系统完全分叉。

## 4. 产品定位

产品定位从：

- 校招简历编辑器

升级为：

- 通用求职简历工具

它仍然服务求职场景，不承担下列定位：

- 个人主页生成器
- 作品集站搭建器
- 自由资料排版工具
- 多用途个人信息平台

允许一定表达性，但表达必须克制、正式、可打印、可投递。

## 5. 信息架构

### 5.1 首页入口

用户进入产品后，先做一次身份轻分流：

- 学生
- 职场人
- 通用求职者

这一步的目的是给用户一个更合适的起点，而不是把产品拆成三套。

### 5.2 编辑器结构

三类入口进入后，仍然进入同一个编辑器内核，统一保留以下能力：

- 模块化填写
- 右侧实时预览
- 模块排序
- 本地保存
- JSON 备份导入导出
- 浏览器打印 / PDF 导出

### 5.3 身份差异承载层

身份差异只体现在以下三层：

- 默认模块
- 默认排序
- 引导文案

除此之外，编辑方式、预览方式、导出链路和底层数据模型尽量统一。

## 6. 身份切换策略

编辑器内允许用户切换身份预设，但采用 `安全切换` 策略。

### 6.1 默认行为

- 切换身份时不覆盖已填写内容
- 不自动重排当前模块顺序
- 不直接替换已有显隐状态
- 只更新推荐提示与推荐配置入口

### 6.2 推荐配置应用

如果用户想套用新的身份推荐配置，则再单独确认后执行，例如：

- 应用该身份推荐模块
- 应用该身份默认排序
- 应用该身份默认引导方式

这样可以避免用户因为一次试探性切换而打乱当前简历。

## 7. 模块体系设计

模块体系采用三层结构：

- 基础必选模块
- 身份差异模块
- 扩展可选模块

### 7.1 基础必选模块

所有身份默认都包含以下共同基础模块：

- 个人信息
- 职业摘要
- 教育经历
- 工作经历
- 项目经历
- 技能
- 证书

这套基础模块用于提供一个全人群通用的求职骨架。

说明：

- 学生可以保留空的工作经历并默认隐藏或弱化
- 职场人仍然保留教育经历，但排序靠后
- 不要求每个用户都完整填写所有共同基础模块

### 7.2 身份差异模块

不同身份在共同基础之上，增加少量默认强化模块。

#### 学生

默认强化模块：

- 校园经历
- 奖项荣誉

#### 职场人

默认强化模块：

- 核心能力
- 个人亮点

#### 通用求职者

不额外增加默认差异模块，保持中性起点。

### 7.3 扩展可选模块

除默认模块外，用户可以从官方模块库中添加更多模块。

第一阶段的模块开放规则：

- 用户可添加官方提供的模块
- 用户不可自由定义任意新模块类型
- 所有模块必须适合正式求职简历 PDF

### 7.4 表达型模块边界

第一阶段允许一定表达性，但表达必须受控。

允许考虑进入模块库的表达型模块示例：

- 个人陈述
- 补充说明
- 兴趣方向

边界要求：

- 篇幅受控
- 不转向个人主页化表达
- 不破坏正式简历的专业感

## 8. 三类身份的默认配置

### 8.1 学生

#### 默认模块

- 共同基础模块
- 校园经历
- 奖项荣誉

#### 默认排序

- 个人信息
- 职业摘要
- 教育经历
- 项目经历
- 校园经历
- 奖项荣誉
- 技能
- 证书
- 工作经历

#### 引导重点

- 强调教育背景和项目实践
- 强调校园经历中的职责与成果
- 强调奖项与能力证明
- 避免只写活动名称或岗位名称

### 8.2 职场人

#### 默认模块

- 共同基础模块
- 核心能力
- 个人亮点

#### 默认排序

- 个人信息
- 职业摘要
- 核心能力
- 工作经历
- 项目经历
- 个人亮点
- 技能
- 证书
- 教育经历

#### 引导重点

- 强调岗位匹配和职业定位
- 强调负责范围、业务场景和结果指标
- 强调项目成果与业务贡献
- 避免流水账式工作描述

### 8.3 通用求职者

#### 默认模块

- 共同基础模块

#### 默认排序

- 个人信息
- 职业摘要
- 教育经历
- 工作经历
- 项目经历
- 技能
- 证书

#### 引导重点

- 提供最稳、最中性的起点
- 鼓励根据目标岗位手动排序
- 强调精简、相关、结果导向

## 9. 引导文案策略

第一阶段的引导设计应保持轻量，不采用重教程模式。

### 9.1 建议形式

- 身份入口的一段开场说明
- 每个重点模块的一句填写提醒
- 必要时提供少量示例表达方式

### 9.2 不建议做法

- 冗长的新手引导流程
- 多步问卷式采集
- 频繁打断编辑操作的提示

目标是让用户感觉“这个工具理解我”，而不是“这个工具在训练我”。

## 10. 数据与配置设计原则

本轮不要求完全重构现有数据结构，但新增设计应遵循以下原则：

- 统一编辑器内核优先
- 身份差异通过配置承载，而不是硬编码多套页面
- 模块显隐、排序和文案尽量可配置
- 共同基础模块与身份差异模块之间边界清晰

建议新增一层预设配置概念，用来表达：

- 身份入口对应的默认模块集合
- 身份入口对应的默认排序
- 身份入口对应的引导文案

## 11. 错误处理与风险控制原则

### 11.1 身份切换风险

风险：

- 用户误操作切换身份后担心内容丢失

原则：

- 默认安全切换
- 需要应用推荐配置时再确认
- 清楚告知“不会覆盖已填写内容”

### 11.2 模块膨胀风险

风险：

- 模块库越做越大，导致上手成本上升

原则：

- 第一阶段只收高频求职模块
- 不引入明显偏作品集站或个人主页的模块
- 每新增模块都要能解释其求职价值

### 11.3 产品定位漂移风险

风险：

- 从通用求职简历工具滑向泛个人资料平台

原则：

- 所有模块必须适合正式求职简历 PDF
- 所有新能力优先服务求职场景
- 不为了“更通用”而破坏简历专业感

## 12. 验收标准

设计落地后，至少应满足以下结果：

1. 用户首次进入时可在三类身份入口中选择起点
2. 三类入口进入后共用同一个编辑器内核
3. 三类入口拥有不同的默认模块、默认排序与引导文案
4. 用户可在编辑器内切换身份，但默认不会覆盖已填写内容
5. 用户可从官方模块库添加扩展模块
6. 所有默认模块与扩展模块都适合导出为正式求职简历 PDF
7. 整体体验仍保持桌面端优先、低门槛、稳定导出

## 13. 推进顺序建议

建议后续实施按以下顺序推进：

1. 将产品重新定义为通用求职简历工具
2. 增加学生 / 职场人 / 通用求职者三类轻分流入口
3. 抽象身份预设配置层
4. 建立共同基础模块、身份差异模块和官方模块库
5. 配置三类身份的默认排序与引导文案
6. 实现安全切换身份机制
7. 再逐步扩展更多高频求职模块

## 14. 为什么选择这条路线

### 14.1 不做多编辑器拆分

原因：

- 维护成本高
- 容易重复造轮子
- 功能、测试、导出逻辑容易分叉
- 不适合当前项目阶段

### 14.2 不做完全自由的通用编辑器

原因：

- 用户起步成本高
- 容易失去当前产品的低门槛优势
- 学生和职场人的默认需求差异无法被有效承接

### 14.3 采用统一内核 + 轻分流 + 官方模块库

收益：

- 最大化复用现有能力
- 明显扩大适用人群
- 保留“好上手”的产品特征
- 为后续继续扩展留足空间

## 15. 结论

下一阶段最合理的产品方向不是：

- 继续只做校招版本
- 或直接做多个独立编辑器
- 或直接做万能个人资料平台

而是：

`把当前项目升级为统一内核的通用求职简历工具，并通过身份轻分流与官方模块库覆盖更广人群。`
